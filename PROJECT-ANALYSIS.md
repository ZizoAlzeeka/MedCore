# منصة كشف التحاليل المكررة — وثيقة التحليل والمهام الكاملة

> **المرحلة:** تخطيط وتحليل (قبل التنفيذ)
> **اللغة:** عربية (RTL) — خط Cairo
> **الغرض:** وثيقة مرجعية شاملة لأي agent أو مطور لفهم المشروع بالكامل والبدء بالتنفيذ.

---

## 1. نظرة عامة على المشروع

### 1.1 الاسم
**نظام كشف التحاليل المكررة (Duplicate Detection System)**

### 1.2 الفكرة
منصة طبية تربط أقسام المستشفى بقاعدة بيانات موحدة، تكشف تلقائياً تكرار التحاليل المخبرية للمريض الواحد، وتُنبه الطبيب قبل تأكيد الطلب. الهدف: منع تكرار الفحوصات غير الضرورية، توفير المال والوقت، وتوحيد السجل الطبي للمريض عبر كل الأقسام والأطباء.

### 1.3 المشكلة
- انعدام الربط الواضح بين الأقسام الطبية.
- المريض يعيد نفس التحليل عدة مرات بدون داعٍ.
- هدر مالي للمريض والمستشفى.
- إرهاق المريض وضياع الوقت.

### 1.4 الحل
نظام **Duplicate Detection** يكشف التحاليل المتكررة بناءً على كود **LOINC** الفريد، ويُنبه الطبيب قبل تأكيد الطلب، مع عرض النتيجة السابقة واتخاذ القرار (تنفيذ / إلغاء / اكتفاء بالسابق).

## 2. التقنيات المستخدمة (Tech Stack)

| الفئة | التقنية | ملاحظات |
|------|---------|---------|
| Backend | **PHP (بدون إطار عمل)** | PHP 8+ مع PDO |
| Database | **MySQL** | عبر PDO Prepared Statements |
| Pattern | **MVC** | Model-View-Controller يدوي |
| Frontend Framework | **Bootstrap 5** (RTL) | من CDN |
| Icons | **Bootstrap Icons** أو Material Icons | من CDN |
| Font | **Cairo** | من Google Fonts |
| Notifications UI | **SweetAlert2** | للتنبيهات والـ popups |
| Rich Text Editor | **Quill** | لكتابة خطة العلاج (مجاني 100% بدون API key) |
| PDF Generation | **html2pdf.js** | لطباعة تقارير التحاليل وخطة العلاج |
| JavaScript | Vanilla JS + jQuery (اختياري مع Bootstrap) | |

### 2.1 قرارات مهمة
- **بدون إطار عمل PHP** (لا Laravel، لا CodeIgniter) — MVC يدوي.
- **بدون تأكيد إيميل** عند التسجيل (لن نستخدم SMTP). الحساب يُفعّل فوراً بعد التسجيل.
- **بدون API keys** خارجية. كل المكتبات مجانية ومستضافة على CDN.

## 4. الأدوار والصلاحيات (5 أدوار)

### 4.1 المدير (Admin)
- إدارة المستخدمين (إضافة/تعديل/تعطيل): أطباء، استقبال، فنيي مختبر، مرضى.
- إدارة الأقسام الطبية: إنشاء قسم، تعديله، إسناد أطباء له.
- إدارة كتالوج التحاليل الشامل (إضافة/تعديل أسماء التحاليل وأكواد LOINC).
- ضبط قواعد كشف التكرار (المدة الزمنية المسموحة بين تحليلين، بالأيام).
- التقارير والإحصائيات: عدد التحاليل الممنوعة، التوفير المالي، أداء الأقسام.
- الإعدادات العامة للنظام.

### 4.2 الطبيب (Doctor)
- تسجيل الدخول بالبريد وكلمة المرور.
- **إدارة جدولة الدوام**: تحديد الأيام والساعات المتاحة للكشف في تواريخ محددة.
- عرض قائمة المرضى المسندين إليه (عبر الاستقبال).
- طلب تحاليل جديدة من الكتالوج.
- استقبال تنبيه التكرار قبل تأكيد الطلب + عرض النتيجة السابقة.
- اتخاذ القرار: تنفيذ / إلغاء / اكتفاء بالسابق.
- عرض التاريخ الطبي الكامل لكل مريض.
- استقبال إشعار عند رفع نتيجة تحليل طلبه.
- **كتابة خطة العلاج**: بعد عودة المريض بنتيجة التحاليل، يفتح صفحة المريض، يكتب اسم العلاج + الوصف + طريقة الاستخدام عبر محرر Quill، يحفظ، فيُعلَم المريض.
- إحالة المريض لطبيب آخر في قسم مغاير.

### 4.3 الاستقبال (Reception)
- تسجيل الدخول بالبريد وكلمة المرور.
- **التحقق من هوية المريض**: بالاسم + الرقم المميز (10 أرقام).
- **تسجيل مرضى جدد** لا يملكون حساباً.
- **حجز موعد للمريض مع الطبيب**: اختيار القسم → اختيار الطبيب → اختيار الموعد المتاح من جدول الدوام.
- **عرض جداول مواعيد الأطباء** (دوام كل طبيب ومواعيده المحجوزة).
- استقبال المراجعين وتوجيههم.
- طباعة بطاقة الموعد.

> **قرار:** الحجز يتم حصراً عبر الاستقبال (لا حجز ذاتي من المريض) لتقليل حجم النظام.

### 4.4 فني المختبر (Lab Technician)
- تسجيل الدخول بالبريد وكلمة المرور.
- **عرض الطلبات الواردة** للمختبر (كل التحاليل المطلوبة من كل الأطباء).
- **رفع قيم/نتائج التحاليل** على ملف المريض بعد تنفيذها مخبرياً.
- إدخال: قيمة النتيجة، الوحدة، النطاق الطبيعي، ملاحظات.
- **عند الرفع**: يصل إشعار تلقائي للمريض + للطبيب الذي طلب التحاليل.
- تتغير حالة الطلب من "بانتظار التنفيذ" إلى "تم رفع النتيجة".
- لا يمكن لفني المختبر طلب تحاليل أو تعديل خطة علاج.

### 4.5 المريض (Patient)
- **إنشاء حساب ذاتي** بالبريد وكلمة المرور + بيانات كاملة (بدون تأكيد إيميل — يُفعّل فوراً).
- توليد **رقم مميز فريد** من 10 أرقام تلقائياً بعد التسجيل.
- لوحة تحكم شخصية.
- عرض التحاليل السابقة والنتائج.
- عرض خطة العلاج المكتوبة من الطبيب.
- استلام إشعارات بنتائج التحاليل الجديدة وخطة العلاج.
- **طباعة تقرير PDF** لنتائج التحاليل وخطة العلاج.
- ملف شخصي قابل للتعديل (البيانات غير الحرجة).
- **لا يمكنه حجز موعد ذاتياً** — الحجز فقط عبر الاستقبال.

## 5. الأقسام الطبية (إدارة المدير)

المدير وحده من يُنشئ الأقسام ويُسند لها الأطباء. أمثلة:
- قسم الباطنة
- قسم القلب
- قسم العيون
- قسم الأطفال
- قسم الجهاز الهضمي
- قسم النساء والولادة

كل قسم له:
- `id`
- `name_ar` (عربي)
- `name_en` (إنجليزي)
- `description`
- الأطباء المسندون إليه (علاقة many-to-many)

عند تسجيل طبيب جديد، يختار المدير قسمه. يظهر الطبيب في قائمة أطباء القسم، ويستقبل المرضى المسندين إليه.

## 6. كتالوج التحاليل ومعيار LOINC

### 6.1 لماذا LOINC وليس ICD-10؟

| المعيار | الاستخدام | مناسب للتحاليل؟ |
|--------|----------|----------------|
| **ICD-10** | تصنيف **الأمراض والتشخيصات** (مثل: J00 زكام، E11 سكري) | ❌ لا |
| **LOINC** | تسمية **التحاليل المخبرية والقياسات السريرية** | ✅ نعم |

- **LOINC** (Logical Observation Identifiers Names and Codes): معيار عالمي، كل تحال له كود فريد (مثل: `6690-2` لصورة الدم الكاملة CBC).
- **ICD-10** يُستخدم في حقل منفصل "التشخيص المبدئي" عند طلب الطبيب للتحاليل.

### 6.2 بنية كتالوج التحاليل

| الحقل | الوصف |
|------|------|
| `id` | المفتاح الأساسي |
| `loinc_code` | كود LOINC الفريد (مثل `6690-2`) |
| `name_ar` | الاسم بالعربي |
| `name_en` | الاسم بالإنجليزي |
| `category` | الفئة (أمراض دم، كيمياء حيوية، وظائف كلى...) |
| `sample_type` | نوع العينة (دم، بلازما، مصل، بول...) |

### 6.3 دور المعيار في كشف التكرار
لو طلب طبيبان نفس التحاليل بكتابة اسمين مختلفين ("CBC" و"صورة دم")، النظام يكتشف أنهما نفس التحاليل لأن لهما نفس كود LOINC. هذا هو أساس خوارزمية كشف التكرار.

### 6.4 أمثلة من الكتالوج

| LOINC | الاسم عربي | الاسم إنجليزي | الفئة | نوع العينة |
|-------|-----------|--------------|------|-----------|
| 6690-2 | صورة دم كاملة | CBC | أمراض الدم | دم وريدي |
| 2345-7 | سكر صائم | Glucose Fasting | كيمياء حيوية | بلازما |
| 2951-2 | صوديوم في الدم | Sodium Serum | كيمياء حيوية | مصل |
| 33914-3 | وظائف الكبد | Liver Panel | كيمياء حيوية | مصل |
| 2160-0 | الكرياتينين | Creatinine | وظائف الكلى | مصل |

## 8. الأقسام البرمجية (Modules)

### 8.1 نظام المصادقة (Auth)
- تسجيل دخول (بريد + كلمة مرور).
- تسجيل المريض ذاتياً (بدون تأكيد إيميل).
- تسجيل الخروج.
- استرجاع كلمة المرور (سؤال أمني أو إعادة تعيين من المدير — بدون SMTP).
- توليد `unique_id` فريد من 10 أرقام عشوائية عند التسجيل.
- تشفير كلمة المرور بـ `password_hash()` (BCRYPT).

### 8.2 لوحات التحكم (Dashboards)
- **Admin**: إحصائيات سريعة، آخر التنبيهات، روابط للإدارة.
- **Doctor**: مواعيد اليوم، آخر الطلبات، تنبيهات غير مقروءة.
- **Reception**: حجز موعد، بحث عن مريض، عرض جداول الأطباء.
- **Lab Tech**: قائمة الطلبات بانتظار التنفيذ.
- **Patient**: آخر التحاليل والنتائج، خطة العلاج، الإشعارات.

### 8.3 إدارة المستخدمين (Admin)
- CRUD كامل لكل دور.
- تفعيل/تعطيل الحسابات.
- إعادة تعيين كلمة المرور.

### 8.4 إدارة الأقسام (Admin)
- CRUD للأقسام.
- إسناد أطباء للأقسام.

### 8.5 كتالوج التحاليل (Admin)
- CRUD للتحاليل مع أكواد LOINC.
- بحث وفلترة.

### 8.6 جدولة دوام الطبيب (Doctor)
- الطبيب يحدد أيام وساعات دوامه.
- الاستقبال يرى الجداول المتاحة عند الحجز.

### 8.7 حجز المواعيد (Reception)
- البحث عن مريض (بالاسم أو الرقم المميز).
- اختيار القسم → الطبيب → الموعد المتاح.
- تأكيد الحجز.

### 8.8 طلب التحاليل (Doctor)
- البحث في الكتالوج.
- اختيار التحاليل للمريض.
- **كشف التكرار تلقائياً قبل الحفظ**.
- عرض التنبيه + النتيجة السابقة.
- القرار: تنفيذ / إلغاء / اكتفاء.

### 8.9 رفع نتائج المختبر (Lab Tech)
- عرض الطلبات `status = 'ordered'`.
- إدخال النتائج (القيمة، الوحدة، النطاق، العلم).
- حفظ → تغيير الحالة إلى `result_uploaded`.
- إرسال إشعار للمريض + الطبيب الطالب.

### 8.10 خطة العلاج (Doctor)
- تظهر فقط بعد رفع نتيجة التحاليل.
- الطبيب يفتح صفحة المريض → "إضافة خطة علاج".
- يكتب: اسم العلاج + الوصف (Quill rich text).
- حفظ → إشعار للمريض.
- المريض يرى خطة العلاج في صفحته.

### 8.11 نظام التنبيهات (Notifications)
- داخل النظام (in-app) — لا إيميل.
- عرض في الـ Dashboard + Badge بعد.
- أنواع: جاهزية النتيجة، إضافة خطة علاج، حجز موعد، تنبيه تكرار.

### 8.12 التقارير والإحصائيات (Admin)
- عدد التحاليل الممنوعة (التي ألغاها الطبيب بسبب التكرار).
- التوفير المالي التقديري.
- أكثر الأطباء طلباً للتحاليل.
- أداء الأقسام.

### 8.13 طباعة PDF (Patient)
- زر "طباعة تقرير التحاليل" في صفحة المريض.
- زر "طباعة خطة العلاج".
- يستخدم **html2pdf.js** لتحويل HTML إلى PDF.

### 8.14 الإحالات (Doctor)
- الطبيب يحوّل المريض لطبيب آخر في قسم مغاير.
- تظهر الزيارة في قائمة الطبيب الجديد.

### 8.15 الملف الشخصي (All roles)
- عرض البيانات الكاملة + الرقم المميز.
- تعديل البيانات غير الحرجة (الهاتف، العنوان).
- لا يمكن تعديل البريد أو الرقم المميز إلا بطلب للمدير.

### 8.16 الإعدادات (Admin)
- ضبط نافذة كشف التكرار (بالأيام).
- اسم الموقع.

## 9. السيناريو الكامل المفصّل

### 9.1 المرحلة الأولى — إنشاء الحساب
1. المريضة "سارة" تدخل صفحة التسجيل، تملأ: الاسم الرباعي، البريد، كلمة المرور (مرئية بكبسة eye)، رقم الموبايل (حقل text مع pattern أرقام إنجليزية)، العنوان، تاريخ الميلاد، الجنس (أنثى).
2. تضغط "تسجيل". الحساب يُفعّل **فوراً بدون تأكيد إيميل**.
3. النظام يولّد لها رقم مميز فريد من 10 أرقام (مثال: `7492036185`).
4. تُوجَّه إلى لوحة تحكم المريض.

### 9.2 المرحلة الثانية — حجز الموعد (عبر الاستقبال)
1. سارة تذهب للمستشفى. موظف الاستقبال يبحث عنها بالاسم أو الرقم المميز.
2. يفتح صفحة "حجز موعد": يختار القسم (باطنة) → يظهر قائمة أطباء القسم مع جدول دوام كل واحد.
3. يختار "د. أحمد علي" → يختار موعداً متاحاً من جدول دوامه (الأحد 10 صباحاً).
4. يأكد الحجز. تُطبَع بطاقة الموعد. الحالة: "بانتظار الكشف".

### 9.3 المرحلة الثالثة — الزيارة الأولى وطلب التحاليل
1. د. أحمد يفتح قائمة مواعيده اليوم، يجد سارة.
2. يفتح ملفها (فارغ حالياً). يفحصها، يشتبه في مشكلة معدية.
3. يضغط "طلب تحاليل جديد"، يبحث عن "CBC" في الكتالوج.
4. النظام يفحص تاريخ سارة: **لا توجد تحاليل سابقة** → لا تنبيه.
5. يحدد التشخيص ICD-10 (R10 — ألم بطني)، يضغط "تأكيد الطلب".
6. الحالة: `ordered`. الطلب يظهر في صفحة فني المختبر.

### 9.4 المرحلة الرابعة — رفع نتيجة المختبر
1. سارة تذهب للمختبر، تُسحب العينة، يُنفَّذ التحاليل.
2. فني المختبر يفتح صفحة "الطلبات الواردة"، يجد طلب سارة.
3. يُدخل النتائج: الهيموجلوبين 12.5 g/dL، كريات بيض 7000، إلخ. يحدد النطاق الطبيعي والعلم (normal/high/low).
4. يضغط "رفع النتيجة". الحالة: `result_uploaded`.
5. **يُرسَل إشعار تلقائياً**: لسارة (في لوحتها) + لد. أحمد (الطبيب الطالب).
6. في صفحة سارة: يظهر التحاليل بحالة "تم رفع النتيجة" + زر "عرض التقرير".
7. **خطة العلاج غير متاحة بعد** — سارة يجب أن تعود للطبيب.

### 9.5 المرحلة الخامسة — الزيارة الثانية وكشف التكرار
1. بعد أسبوع، تعود أعراض سارة. تذهب للاستقبال، تحجز موعداً مع "د. محمد نور" (نفس قسم الباطنة).
2. د. محمد يفتح ملفها، يرى الزيارة السابقة والتحاليل المنفذ.
3. يريد طلب "صورة دم كاملة" للتأكد. يبحث في الكتالوج، يختار CBC.
4. **النظام يكشف التكرار!** يُجد أن تحاليل CBC (نفس كود LOINC 6690-2) تم تنفيذه قبل 7 أيام.
5. يظهر تنبيه SweetAlert2: "تنبيه: هذا التحاليل تم إجراؤه مؤخراً بتاريخ 04/07/2026 — قبل 7 أيام. اضغط لعرض النتيجة السابقة."
6. د. محمد يضغط "عرض النتيجة السابقة"، يرى القيم الكاملة.
7. يقرر "الاكتفاء بالنتيجة السابقة". يُسجَّل القرار في `duplicate_alerts`. الحالة: `duplicate_skipped`.
8. يظهر القرار في تقرير المدير: تحاليل ممنوع + التوفير المالي.

### 9.6 المرحلة السادسة — كتابة خطة العلاج
1. د. محمد يضغط "إضافة خطة علاج" في صفحة سارة.
2. يكتب: اسم العلاج "أوميبرازول 20mg" + الوصف (عبر Quill editor: "يُؤخَذ قرص واحد صباحاً قبل الفطور لمدة 14 يوم، مع تجنب الأطعمة الحارة").
3. يحفظ. يُرسَل إشعار لسارة: "تمت إضافة خطة علاج جديدة".
4. سارة تفتح صفحتها، ترى خطة العلاج منسّقة (bold، قوائم، إلخ).
5. تضغط "طباعة PDF" → يُولَّد ملف PDF عبر html2pdf.js.

### 9.7 المرحلة السابعة — الإحالة (اختياري)
1. د. محمد يقرر تحويل سارة لقسم الجهاز الهضمي.
2. يضغط "إحالة" → يختار الطبيب المناسب.
3. تظهر زيارة سارة في قائمة الطبيب الجديد، مع كامل تاريخها الطبي.
4. الطبيب الجديد يطّلع على كل شيء دون إعادة أي تحاليل.

## 10. قواعد العمل (Business Rules)

### 10.1 قواعد كشف التكرار
- النظام يفحص: نفس `patient_id` + نفس `loinc_code` (عبر `test_id` المرتبط) + ضمن نافذة زمنية (افتراضي 30 يوم، قابلة للتعديل من الإعدادات).
- الفحص يتم **قبل حفظ الطلب**، فيظهر التنبيه للطبيب ليقرر.
- قرار الطبيب يُسجَّل في `duplicate_alerts.doctor_decision`.

### 10.2 قواعد الحجز
- الحجز فقط عبر الاستقبال (لا حجز ذاتي للمريض).
- لا يمكن حجز موعد في فترة غير متاحة في جدول دوام الطبيب.
- لا يمكن حجز موعدين متداخلين لنفس المريض.

### 10.3 قواعد رفع النتائج
- فقط فني المختبر يمكنه رفع النتائج.
- لا يمكن رفع نتيجة لطلب ملغى أو مكرر متخطى.
- عند الرفع: يتغير `status` إلى `result_uploaded` + إشعار للمريض والطبيب.

### 10.4 قواعد خطة العلاج
- تظهر فقط بعد `status = result_uploaded` (لا يمكن كتابتها قبل رفع النتائج).
- فقط الطبيب المعالج يكتبها.
- يمكن تعديلها لاحقاً (يُحفَظ تاريخ التعديل).

### 10.5 قواعد الرقم المميز
- 10 أرقام عشوائية، فريدة عالمياً في جدول `users`.
- يُولَّد تلقائياً عند التسجيل (لا يدخله المستخدم).
- لا يمكن تعديله إلا بطلب للمدير.
- يُستخدم للتحقق من الهوية في الاستقبال.

### 10.6 قواعد الحسابات
- **بدون تأكيد إيميل** — الحساب يُفعّل فوراً بعد التسجيل.
- كلمة المرور مشفّرة بـ `password_hash()` (BCRYPT).
- لا يمكن تعديل البريد إلا بطلب للمدير.
- الحساب المعطّل لا يمكنه الدخول.

## 11. متطلبات الواجهة (UI/UX)

### 11.1 عام
- اللغة: عربية فقط، اتجاه `dir="rtl"`.
- الخط: **Cairo** (Google Fonts) لكل النصوص.
- الإطار: **Bootstrap 5 RTL**.
- الألوان: تدرّجات linear-gradient بنفسجي (#6C63FF) + وردي (#FF6584) + أزرق (#4FC3F7) + سماوي (#81D4FA).
- التنبيهات والـ popups: **SweetAlert2** (بدل alert العادي).
- الأيقونات: **Bootstrap Icons**.

### 11.2 حقول كلمة المرور (Password Fields)
كل حقل password في النظام يجب أن:
- يكون نوعه `type="password"`.
- يوجد بجانبه **زر eye** (أيقونة عين) لتبديل الإظهار/الإخفاء.
- عند الضغط: يتبدل النوع بين `password` و `text`، وتبدل الأيقونة بين `bi-eye` و `bi-eye-slash`.
- يستخدم Bootstrap input group:
```html
<div class="input-group">
  <input type="password" id="password" name="password" class="form-control">
  <button class="btn btn-outline-secondary" type="button" onclick="togglePassword('password', this)">
    <i class="bi bi-eye"></i>
  </button>
</div>
```
```javascript
function togglePassword(fieldId, btn) {
  const input = document.getElementById(fieldId);
  const icon = btn.querySelector('i');
  if (input.type === 'password') {
    input.type = 'text';
    icon.classList.replace('bi-eye', 'bi-eye-slash');
  } else {
    input.type = 'password';
    icon.classList.replace('bi-eye-slash', 'bi-eye');
  }
}
```

### 11.3 حقول الأرقام (Number Fields)
كل حقل رقمي (رقم الموبايل، العمر، الرقم المميز، إلخ) يجب أن:
- يكون `type="text"` **وليس** `type="number"` (لتجنب ظهور أرقام عربية-هندية ٠١٢٣ في بعض المتصفحات العربية).
- يضاف `inputmode="numeric"`.
- يضاف `pattern="[0-9]*"` (أو نمط أدق مثل `pattern="[0-9]{10}"` للموبايل).
- يُستخدم JavaScript لمنع إدخال غير الأرقام الإنجليزية:
```html
<input type="text" inputmode="numeric" pattern="[0-9]*"
       oninput="this.value=this.value.replace(/[^0-9]/g,'')"
       class="form-control" name="phone">
```

### 11.4 محرر خطة العلاج (Quill)
- يُحمَّل في صفحة "إضافة خطة علاج" للطبيب.
- يدعم: bold، italic، underline، lists (مرتبة وغير مرتبة)، headings، alignment، blockquote.
- الاتجاه RTL.
- عند الحفظ: يُؤخَذ `quill.root.innerHTML` ويُحفَظ في `description_html` (LONGTEXT).
- عند العرض: يُحقَن الـ HTML في div مع sanitization (htmlspecialchars_decode).

```javascript
const quill = new Quill('#editor-container', {
  theme: 'snow',
  rtl: true,
  modules: {
    toolbar: [
      [{ 'header': [1, 2, 3, false] }],
      ['bold', 'italic', 'underline'],
      [{ 'list': 'ordered'}, { 'list': 'bullet' }],
      [{ 'align': [] }],
      ['blockquote'],
      ['clean']
    ]
  }
});
```

### 11.5 طباعة PDF
- زر "طباعة التقرير" في: صفحة المريض (تقرير التحاليل)، صفحة خطة العلاج.
- يُنشئ HTML مخفياً بتنسيق مناسب (A4، هوامش، خط Cairo)، ثم يمرره لـ html2pdf.js:
```javascript
function printReport(elementId, filename) {
  const element = document.getElementById(elementId);
  html2pdf().set({
    margin: 10,
    filename: filename,
    image: { type: 'jpeg', quality: 0.98 },
    html2canvas: { scale: 2 },
    jsPDF: { unit: 'mm', format: 'a4', orientation: 'portrait' }
  }).from(element).save();
}
```

### 11.6 لوحات التحكم
- **Admin**: بطاقات إحصائية + جدول آخر التنبيهات + روابط سريعة.
- **Doctor**: مواعيد اليوم (cards) + تنبيهات غير مقروءة (badge) + آخر الطلبات.
- **Reception**: نموذج بحث عن مريض + زر "حجز موعد جديد" + جدول مواعيد اليوم.
- **Lab Tech**: قائمة الطلبات `status='ordered'` مع زر "رفع نتيجة".
- **Patient**: آخر التحاليل (cards) + خطة العلاج + الإشعارات + زر طباعة PDF.

## 14. ملاحظات للمطوّر / Agent

### 14.1 قرارات معتمدة (لا تحتاج نقاش)
1. ✅ **LOINC** معتمد لتسمية التحاليل.
2. ✅ **الحجز عبر الاستقبال فقط** (لا حجز ذاتي للمريض) — لتقليل حجم النظام.
3. ✅ **بدون تأكيد إيميل** (لا SMTP) — الحساب يُفعّل فوراً.
4. ✅ **5 أدوار**: admin, doctor, reception, lab_tech, patient.
5. ✅ **خطة العلاج** تُكتَب بعد رفع النتائج فقط، من قبل الطبيب، عبر Quill.
6. ✅ **فني المختبر** هو من يرفع النتائج (ليس الطبيب).
7. ✅ **بدون API keys** خارجية — كل المكتبات مجانية.

### 14.2 نقاط يجب الانتباه لها
- **الرقم المميز**: 10 أرقام عشوائية، تحقق من التفرّد قبل الحفظ.
- **حقول الأرقام**: دائماً `type="text"` + `pattern="[0-9]*"` + `inputmode="numeric"` لتفادي الأرقام العربية-الهندية.
- **حقول كلمة المرور**: دائماً مع زر eye للتبديل.
- **Quill HTML**: يجب sanitization عند العرض لمنع XSS (استخدم library مثل HTML Purifier أو strip_tags للوسوم الخطرة).
- **التنبيهات**: كلها in-app (لا إيميل، لا SMS).
- **كشف التكرار**: يحدث **قبل** حفظ الطلب، في نفس الـ request (AJAX أو بعد submit).
- **Bootstrap RTL**: استخدم `bootstrap.rtl.min.css` وليس `bootstrap.min.css`.
- **SweetAlert2**: استبدل كل `alert()` و `confirm()` العادية.

### 14.3 ترتيب التنفيذ المقترح
1. المجموعة A (تأسيس) → B (مصادقة) → C (لوحات).
2. D (إدارة المدير) → ضرورية قبل أي شيء (بدون أقسام وأطباء وكتالوج، لا يعمل النظام).
3. E (جدولة) → F (حجز) → G (طلب تحاليل + كشف تكرار).
4. H (رفع نتائج) → I (خطة علاج) → J (إشعارات).
5. K (طباعة PDF) → L (إحالات) → M (أمان).
